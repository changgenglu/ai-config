# Critical Analysis Skill

你剛完成了一份產出。現在切換到批判者視角，對自己的產出進行多維度審查。

## 審查維度

### 1. 假設檢驗

- 產出中有哪些隱含假設？逐一列出。
- 每個假設如果不成立，會造成什麼後果？
- 哪些假設需要使用者確認？

### 2. 遺漏檢查

- 有哪些邊界條件沒有覆蓋？
- 有哪些錯誤情境（error path）沒有考慮？
- 有哪些角色/權限組合沒有處理？

### 3. 替代方案

- 有沒有更簡單的做法被忽略了？
- 目前方案的最大風險是什麼？
- 如果要推翻目前方案，最強的理由是什麼？

### 4. 依賴與影響

- 這個方案會影響哪些現有功能？
- 有沒有隱性的資料依賴？
- 效能影響評估是否完整？

## 輸出格式

將審查結果附在你的主要產出末尾：

```
## 批判性審查

### 隱含假設
- [假設內容] → 若不成立：[後果] → 狀態：[已確認 / 待確認]

### 遺漏風險
- [風險描述] → 嚴重程度：[高/中/低]

### 替代方案
- [替代做法] → 未採用原因：[原因]（若無明確原因則標記「值得討論」）

### 影響範圍
- [受影響的模組/功能]
```

## 使用原則

- 不要為了找問題而硬找——如果某個維度確實沒有問題，寫「無發現」
- 聚焦在可行動的發現，不做空泛的警告
- 標記「待確認」的項目會成為停頓點中使用者需要回答的問題
